# Zac v5 要件定義

作成日: 2026-04-27  
基準文書: `Zac_planning_v5.md`

---

## 1. プロダクト目的

Zacは、クライマーが「次に登る場所・時間・仲間」を見つけ、登った体験を記録し、共有し、次のセッションにつなげるための Climb Life OS である。

MVPでは、クライミングSNS全体を作るのではなく、次の循環を成立させる。

```text
ジムを探す
  -> 保存する
  -> 行く予定を作る
  -> 登る
  -> 記録する
  -> 投稿/反応が生まれる
  -> 次の予定を作る
```

---

## 2. MVPの成功条件

MVPは次の状態を満たしたら成功とみなす。

- 登録後、ユーザーがオンボーディングを完了できる。
- ユーザーがジムを検索・保存できる。
- ユーザーが保存ジムまたは任意場所で予定を作成できる。
- ユーザーが予定に参加・コメントできる。
- ユーザーが予定から記録を作成できる。
- ユーザーが記録を作成・編集・削除できる。
- ユーザーが記録から投稿を作成できる。
- フィードに予定、記録、投稿が混在表示される。
- ブロック、通報、非表示が最低限動作する。
- 管理者がジム、投稿、通報を管理できる。
- 重大な公開範囲漏れがない。

---

## 3. ユーザー種別

| 種別 | 説明 | 主な権限 |
|---|---|---|
| 未ログインユーザー | 登録前の閲覧者 | 公開ジム、公開イベントの閲覧 |
| 一般ユーザー | Zacの通常利用者 | 予定、記録、投稿、保存、フォロー、通報 |
| 管理者 | 運営担当者 | ユーザー/投稿/ジム/イベント/通報管理 |
| 停止ユーザー | 規約違反等で制限されたユーザー | ログイン後の操作制限 |

MVPではジム公式アカウントは作らない。ジム情報は管理者が登録・更新する。

---

## 4. 機能要件

### 4.1 認証

- メール/パスワード登録ができる。
- OAuthログインはMVPでは実装しない。Web版検証後にGoogle OAuthから追加する。
- ログアウトできる。
- パスワードリセットができる。
- 退会できる。
- 退会時のデータ削除/匿名化方針を適用する。

### 4.2 プロフィール/オンボーディング

- 表示名、アイコン、自己紹介を設定できる。
- 主な種目を選択できる。
- クライミング歴を選択できる。
- よく行くエリアを設定できる。
- 興味カテゴリを選択できる。
- 初期公開範囲を選択できる。
- 位置情報利用は任意で、デフォルトOFFにする。

### 4.3 ジム探索

- ジム一覧を閲覧できる。
- ジム名、エリア、種目で検索できる。
- ジム詳細を閲覧できる。
- ジムを保存できる。
- ジム詳細から予定を作成できる。
- 管理者はジムを作成・編集・公開/非公開にできる。

### 4.4 予定

- ユーザーは行く予定を作成できる。
- 予定には開始/終了時刻、場所、種目、公開範囲、募集状態を持たせる。
- 予定は `draft`, `scheduled`, `active`, `completed`, `cancelled` の状態を持つ。
- ユーザーは参加/キャンセルできる。
- 予定にコメントできる。
- 予定完了後、記録へ変換できる。
- 公開範囲、ブロック、参加状態に応じて表示可否を制御する。

### 4.5 記録

- ユーザーは日付、場所、種目、グレード、メモを記録できる。
- 予定から記録を作成できる。
- 記録に画像を追加できる。
- 記録を編集・削除できる。
- 記録から投稿を作成できる。
- 非公開ログを許可する。

### 4.6 投稿/フィード

- ユーザーはテキスト/画像投稿を作成できる。
- 投稿は単独投稿、記録由来、予定由来、ジム由来を区別する。
- 投稿にいいね、コメント、保存ができる。
- フィードには予定、記録、投稿、ジム更新、イベントを混在表示する。
- フィードではブロック、削除、通報中、公開範囲外のコンテンツを除外する。

### 4.7 ソーシャル/安全

- ユーザーをフォロー/フォロー解除できる。
- ユーザーをブロック/ブロック解除できる。
- 投稿、コメント、予定、記録を通報できる。
- ブロック関係にあるユーザーのコンテンツは相互に表示しない。
- DMはMVPでは実装しない。

### 4.8 通知

- 予定参加、コメント、フォロー、ジム/イベント更新を通知する。
- MVPではWebアプリ内通知を優先し、PushはWeb版検証後に段階導入する。

### 4.9 管理画面

- 管理者はユーザーを検索・確認できる。
- 管理者は投稿/コメントを非表示・削除できる。
- 管理者はジム/イベントを作成・編集できる。
- 管理者は通報を確認し、対応ステータスを更新できる。
- 管理者操作は監査ログに残す。

---

## 5. 非機能要件

| 項目 | 要件 |
|---|---|
| セキュリティ | 認証必須API、RBAC、公開範囲制御、ブロック反映を徹底する |
| プライバシー | 位置情報、予定、ホームジム、参加者情報を高リスク情報として扱う |
| 可用性 | MVPでは単一リージョン運用でよいが、DBバックアップを必須にする |
| 性能 | フィード、ジム検索、予定一覧はページネーション必須 |
| 保守性 | TypeScriptで型を共有し、OpenAPI/スキーマを単一ソースに近づける |
| 監査性 | 管理者操作、モデレーション、削除は履歴を残す |
| 拡張性 | 将来のルートDB、ジムアカウント、課金を追加できる余白を残す |

---

## 6. MVP対象外

- フルルートDB/トポ
- ランキング
- 決済
- 会員証
- 予約管理
- 高度なAIコーチング
- 外岩ガイドブック
- DM
- ジム向けSaaS
- 自然言語予定入力

---

## 7. 優先順位

### Must

- Auth/User
- Profile/Onboarding
- Gym Discovery
- Session Plans
- Logs
- Posts/Feed
- Follow/Block/Report
- Admin最低限

### Should

- Webアプリ内通知
- ジムイベント
- 画像複数枚
- 投稿カテゴリ
- 予定リマインダー

### Could

- 投稿文補助AI
- タグ提案
- 詳細統計
- カレンダー連携

### Won't for MVP

- 課金
- 予約
- 会員証
- ルートDB
- ランキング
- OAuthログイン
- Push通知
