# Zac v5 管理画面・運営設計

作成日: 2026-04-27  
基準文書: `Zac_planning_v5.md`

---

## 1. 管理画面の目的

MVPの管理画面は、プロダクト運営に必要な最低限の安全管理とデータ整備を行う。

優先順位:

1. 通報対応
2. 投稿/コメント非表示
3. ジム情報管理
4. ユーザー状態管理
5. 監査ログ確認

---

## 2. 管理者ロール

MVPでは1種類の `admin` で開始する。将来、以下に分割する。

| ロール | 役割 |
|---|---|
| `super_admin` | 全権限 |
| `moderator` | 通報、投稿、コメント対応 |
| `gym_editor` | ジム/イベント編集 |
| `support` | ユーザー問い合わせ確認 |

---

## 3. 管理画面機能

| 機能 | 内容 | MVP |
|---|---|---:|
| ダッシュボード | 通報数、登録数、投稿数、予定数 | Must |
| ユーザー管理 | 検索、状態確認、停止、警告 | Must |
| 投稿管理 | 一覧、検索、非表示、削除 | Must |
| コメント管理 | 通報コメント確認、削除 | Must |
| ジム管理 | 登録、編集、画像、公開/非公開 | Must |
| イベント管理 | 登録、編集、公開/非公開 | Should |
| 通報管理 | 一覧、詳細、対応ステータス | Must |
| モデレーション履歴 | 誰が何をしたか | Must |
| 告知管理 | アプリ内お知らせ | Should |

---

## 4. 通報対応フロー

```text
通報発生
  -> reportsに保存
  -> 管理画面の通報一覧に表示
  -> 管理者が対象コンテンツを確認
  -> 対応を選択
     - 非表示
     - 削除
     - ユーザー警告
     - 一時停止
     - 永久停止
     - 却下
  -> moderation_actionsに保存
  -> audit_logsに保存
  -> 必要に応じてユーザー通知
```

---

## 5. 通報カテゴリ

- ハラスメント
- スパム
- 不適切画像
- 危険行為の助長
- 個人情報の晒し
- 著作権侵害
- なりすまし
- その他

---

## 6. モデレーションアクション

| action | 内容 |
|---|---|
| `hide_post` | 投稿非表示 |
| `delete_comment` | コメント削除 |
| `warn_user` | ユーザー警告 |
| `suspend_user` | 一時停止 |
| `ban_user` | 永久停止 |
| `dismiss_report` | 通報却下 |
| `mark_review_pending` | 管理者レビュー待ち |

---

## 7. ジム情報更新フロー

```text
ジム情報追加/修正
  -> 管理者が情報確認
  -> 住所/種目/画像/営業時間を登録
  -> draftで保存
  -> 内容確認
  -> publishedへ変更
  -> ユーザーが保存/予定作成可能になる
```

---

## 8. 監査ログ

以下は必ず監査ログに残す。

- 管理者ログイン
- ユーザー停止/解除
- 投稿非表示/復元
- コメント削除
- 通報ステータス変更
- ジム作成/更新/公開/非公開
- イベント作成/更新/公開/非公開

監査ログ項目:

- actor_id
- action
- target_type
- target_id
- metadata
- created_at

---

## 9. 運営KPI

- 未対応通報数
- 通報対応時間
- 非表示件数
- ユーザー停止件数
- ジム情報更新数
- ジム情報修正依頼数

---

## 10. 固定事項

- 管理者アカウントはSupabase Authで作成し、DB上の管理者ロールに紐づける。
- 管理者2FAはベータ/本番運用前に必須にする。
- 通報対応SLAはP0即時、P1当日、P2次回リリース候補、P3通常バックログとする。
- ユーザー警告文テンプレートは管理画面実装時に初期文言をseedする。
- ジム情報は公開情報または許諾済み情報のみ登録し、更新元を管理者メモに残す。
