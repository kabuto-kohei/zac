# Zac v5 リリース・運用計画書

作成日: 2026-04-27  
基準文書: `Zac_planning_v5.md`

---

## 1. 目的

MVP公開後に、障害、通報、データ更新、問い合わせに対応できる最低限の運用体制を決める。

---

## 2. リリース段階

| 段階 | 内容 | 判定 |
|---|---|---|
| internal alpha | 開発者/関係者のみ | 基本導線が通る |
| closed beta | 20〜50人 | 予定と記録が自然に使われる |
| public beta | 一般登録開始 | 重大な安全問題がない |
| production | 正式運用 | KPIと運用負荷を確認後 |

---

## 3. リリース手順

1. migrationを確認する。
2. seedデータを確認する。
3. API health checkを確認する。
4. 管理画面ログインを確認する。
5. User Web/Admin/APIのbuildを作成する。
6. 主要E2Eを実行する。
7. Sentry/PostHogの疎通を確認する。
8. リリースノートを書く。
9. 配布する。
10. リリース後24時間のエラーと通報を監視する。

---

## 4. ロールバック方針

- API/Adminは前バージョンへ戻せるようにする。
- User Webは前バージョンへ戻せるようにする。
- DB migrationは破壊的変更を避ける。
- Web/API/Adminで問題が出た場合はVercelの前バージョンへ戻す。
- モバイル版は後続フェーズのため、ストア配布時のロールバック方針は別途定義する。
- 問題がある機能はサーバー側で非表示/停止できるようにする。

---

## 5. 監視

| 対象 | ツール | 見るもの |
|---|---|---|
| API | Sentry/logs | 5xx、latency、例外 |
| User Web | Sentry | browser error、network error |
| Admin | Sentry | 操作エラー |
| Product | PostHog | signup、plan、log、post |
| DB | hosting metrics | CPU、connection、slow query |

---

## 6. 障害対応

| レベル | 内容 | 対応 |
|---|---|---|
| P0 | 個人情報/非公開予定の漏えい | 即時停止、原因調査、通知判断 |
| P1 | 登録/ログイン/予定/記録が使えない | 当日対応 |
| P2 | 投稿/通知/画像の一部不具合 | 次回リリース候補 |
| P3 | 表示崩れ、軽微な文言 | 通常バックログ |

---

## 7. バックアップ

- DBバックアップを日次で取得する。
- 重要リリース前に手動バックアップを取得する。
- 復旧手順を検証する。
- 削除済みUGCの画像は通常APIで返さず、非同期削除対象にする。

---

## 8. 問い合わせ対応

MVPでは簡易フォームまたはメールで受ける。

- ログインできない
- 投稿/画像削除依頼
- ジム情報修正依頼
- 通報/安全問題
- バグ報告
- その他

安全問題、個人情報、未成年関連は優先対応する。

---

## 9. 運用上の禁止事項

- 本番DBを直接編集しない。
- 秘密情報をチャットやIssueに貼らない。
- 本番ユーザーデータをローカルへ不用意にコピーしない。
- 通報対応を監査ログなしで行わない。
