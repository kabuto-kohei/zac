# Zac v5 プロジェクト計画書

作成日: 2026-04-27  
基準文書: `Zac_planning_v5.md`

---

## 1. 目的

Zac MVPを、スコープを膨らませずに「予定 -> 記録 -> 次の予定」の検証ができる状態まで開発する。

---

## 2. スコープ

### MVPに含める

- 認証/退会
- プロフィール/オンボーディング
- ジム一覧/詳細/保存
- 予定作成/参加/コメント/完了
- 記録作成/編集/削除
- 記録から投稿
- 投稿/フィード/いいね/コメント/保存
- フォロー/ブロック/通報
- 管理画面
- KPI計測

### MVPに含めない

- ルートDB
- 決済
- 会員証
- 予約
- ジムSaaS
- DM
- ランキング
- 高度AIコーチング

---

## 3. マイルストーン

| Phase | 内容 | 完了条件 |
|---|---|---|
| Phase 0 | 設計固定 | docs一式、技術選定、DB/API初版 |
| Phase 1 | MVP基盤 | 認証、ユーザー、ジム、管理画面基盤 |
| Phase 2 | Core Loop | 予定、記録、予定から記録 |
| Phase 3 | Social/Safety | 投稿、フィード、フォロー、ブロック、通報 |
| Phase 4 | Closed Beta | 20〜50人で検証 |
| Phase 5 | Public Beta | 一般登録開始 |

---

## 4. 実装優先順位

1. 認証/ユーザー
2. ジム
3. 予定
4. 記録
5. 投稿/フィード
6. ブロック/通報/管理
7. 通知/計測

---

## 5. 役割

| 役割 | 責任 |
|---|---|
| Product Owner | スコープ、KPI、優先順位の決定 |
| Web Engineer | User Web実装 |
| Backend Engineer | API/DB/Auth/Storage |
| Admin Engineer | 管理画面 |
| Designer | 画面設計、UI仕様 |
| QA | テスト、受け入れ確認 |
| Operator | 通報、ジム情報、問い合わせ対応 |

---

## 6. 主要リスク

| リスク | 対策 |
|---|---|
| スコープ肥大 | MVP対象外をIssue化しすぎない |
| 公開範囲漏れ | visibility serviceを共通化しテスト必須 |
| 初期データ不足 | 3〜5ジムを運営登録 |
| 投稿荒れ | 通報/非表示/ブロックをMVPに含める |
| 記録が面倒 | 最小入力だけで保存可能にする |
| 技術選定遅れ | Auth/API/ORM/Storageを先に固定 |

---

## 7. 変更管理

- `session_plans` と `climbing_logs` の中核価値に直結する変更は検討する。
- ルートDB、課金、会員証、予約、DMはMVP後に回す。
- プライバシー/安全に関わる変更は優先する。
- DB破壊的変更、本番データ削除、公開範囲変更は明示承認を必要とする。

---

## 8. 完了判定

MVPは、次を満たしたらクローズドベータへ進める。

- 最初の縦スライスが通っている。
- 予定から記録への変換が動く。
- 公開範囲、ブロック、通報が最低限動く。
- 管理画面で通報とジムを扱える。
- KPIイベントが計測される。
- P0/P1相当の既知不具合がない。
